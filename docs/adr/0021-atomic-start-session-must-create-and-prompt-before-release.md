# `start-session` must atomically create, prompt, and release before cross-process recovery

Claude and Codex ACP adapters reject recovery of a session that has not completed at least one Agent Turn. In an earlier CLI prototype, the Gateway created a durable session through `create-session`, released the local adapter process, and then attempted recovery from a new process. These adapters returned `Resource not found`: the session was initialized and configured, but no Agent Turn established recoverable conversation state at the adapter level.

## Consequences

- The Gateway exposes `start-session` as the recommended CLI entry for cross-process durable sessions. It creates a durable session, executes the first Agent Turn, and releases local adapter resources atomically within a single process — ensuring the adapter has established recoverable context before the process exits.
- Low-level durable `createSession({ durable: true })` is retained in the TypeScript API for instrumentation, capability inspection, and same-process stateful workflows. The CLI does not expose `create-session`, because releasing an unprompted session produces a Session Reference that cannot be recovered across all verified adapters.
- `start-session` validates its input through `CliRunInputSchema`, the same schema used by the stateless `run` command. Only session-level fields (`apiVersion`, `model`, `permissionPolicy`, `timeoutMs`, `idleTimeoutMs`) are forwarded to `createSession`; prompt fields (`prompt`, `gracePeriodMs`, `includeEvents`) are separated and passed to `session.prompt()`.
- When the first Agent Turn fails, `start-session` attempts to delete the durable record through `forget()` to prevent unrecoverable blank-session references. A failed `start-session` output includes `sessionRef` so consumers can inspect the reference. If `forget()` itself fails (e.g., I/O or permission error), the record may persist and a `cleanup_failed` event is emitted on stderr so consumers can diagnose and manually remove it.
- Real adapter recovery smoke tests use `start-session` as their first step. All three test suites (OpenCode, Claude, Codex) passed through the `start-session` → cross-process recovery → context verification → close lifecycle after this atomic execution model replaced the previous `create-session` → cross-process `prompt` pattern.
- Business Consumers performing multi-process workflows should use `start-session` for the initial turn and `prompt` for subsequent turns. Both commands internally call `release()` after each Agent Turn, preserving the reusable Session Reference.
